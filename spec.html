Visión general

RocketDrop es una aplicación web de página única (SPA) que permite subir archivos de hasta 25 GB directamente a Cloudflare R2 mediante la API de Multipart Upload compatible con S3.

El navegador divide el archivo en fragmentos de 100 MB y los sube concurrentemente (hasta 4 al mismo tiempo), evitando las limitaciones de memoria y tiempo de espera de una carga tradicional de una sola solicitud.

El servidor Express nunca recibe los bytes del archivo: solo orquesta URLs prefirmadas. Los datos van directamente del navegador a R2, manteniendo el servidor ligero y la transferencia rápida.

API Backend

Todos los endpoints aceptan y devuelven application/json. El servidor nunca recibe bytes de archivo.

POST /api/upload/init

Inicia una sesión de carga multiparte en R2.

REQUEST 
{ "filename": "video.mp4", "contentType": "video/mp4" }
RESPONSE 
{ "uploadId": "abc123", "key": "uploads/uuid-video.mp4" }

Llama a CreateMultipartUploadCommand. La clave tiene el formato uploads/{uuid}-{filename}.

POST /api/upload/part-url

Devuelve una URL prefirmada para que el navegador haga PUT de un fragmento directamente a R2.

REQUEST 
{ "key": "uploads/uuid-video.mp4", "uploadId": "abc123", "partNumber": 1 }
RESPONSE 
{ "url": "https://...r2.cloudflarestorage.com/...?X-Amz-Signature=..." }

Usa getSignedUrl + UploadPartCommand. URL válida por 1 hora.

POST /api/upload/complete

Finaliza la carga multiparte después de confirmar todas las partes.

REQUEST 
{ "key": "...", "uploadId": "...", "parts": [{ "PartNumber": 1, "ETag": "\"etag1\"" }] }
RESPONSE 
{ "url": "https://pub-xxx.r2.dev/uploads/uuid-video.mp4" }

Llama a CompleteMultipartUploadCommand. La URL pública es ${R2_PUBLIC_URL}/${key}.

DELETE /api/upload/abort

Cancela una carga multiparte y elimina los datos parciales de R2.

REQUEST 
{ "key": "uploads/uuid-video.mp4", "uploadId": "abc123" }
RESPONSE 
{ "success": true }

Componentes

DropZone src/components/DropZone.tsx

Props: onFile(file: File) => void

Renderiza el área de destino. Gestiona dragover / dragleave para el estado visual, drop para extraer el archivo y un <input type="file"> como fallback. Valida que el archivo no esté vacío y no supere 25 GB.

UploadPanel src/components/UploadPanel.tsx

Props: file, progress, error, onCancel, onReset

Activo durante las fases uploading y error. Muestra nombre y tamaño del archivo, barra de progreso animada, velocidad, ETA, contador de partes, y caja de alerta de error. Incluye botón de cancelar (durante subida) y reiniciar (en error).

ResultPanel src/components/ResultPanel.tsx

Props: phase, file, url, onReset

Activo tras finalizar la carga. Muestra ✅ en éxito o ⛔ en cancelación. En éxito: URL en caja monoespaciada con botón de copia y enlace "Abrir archivo".

Hook principal: src/hooks/useUpload.ts

Flujo de upload(file)

1. phase = 'uploading', guardar archivo en estado
2. POST /api/upload/init → { uploadId, key }
3. Dividir archivo en N partes: Math.ceil(size / CHUNK_SIZE)
4. Por cada parte (Semáforo, máx. 4 concurrentes):
   a. POST /api/upload/part-url → URL prefirmada
   b. xhrPutWithRetry(url, blob, onProgress) → ETag
   c. Actualizar partBytes[i] para cálculo de progreso
   d. Incrementar completedParts
5. POST /api/upload/complete con todos los ETags
6. phase = 'success', resultUrl = URL pública

Cálculo de progreso

totalSubido = sum(partBytes)
pct          = totalSubido / file.size * 100

Por tick de RAF:
  speed = α × (Δbytes / Δt) + (1 - α) × velocidadAnterior   // α = 0.3 EMA
  eta   = (file.size - totalSubido) / speed

Modelo de seguridad

AmenazaMitigación
Credenciales de R2Almacenadas en .env, nunca expuestas al cliente
Mal uso de URL prefirmadaTTL de 1 hora limita la ventana de uso indebido
Archivos demasiado grandesValidación en cliente (UX); sin validación en servidor
Tipo de contenido arbitrarioPasado desde el cliente; R2 acepta cualquier valor
Colisión de claves / path traversalPrefijo UUID previene colisiones y traversal
Acceso sin autenticación⚠️ La app no tiene capa de autenticación
⚠️
No hay limitación de velocidad ni cuota de carga. Cualquier cliente con acceso de red puede subir archivos. Considera añadir autenticación y rate limiting antes de desplegar en un entorno no confiable.

Build y despliegue

Desarrollo

cp .env.example .env        # Completar credenciales de R2
npm install
npm run dev                  # Vite (:5173) + Express (:3000) concurrentes

Vite redirige /api/* a Express para que ambos operen desde el mismo origen en el navegador.

Producción

npm run build                # tsc + vite build → dist/
npm start                    # Express sirve dist/ + rutas /api/* en PORT

Modo producción de Express

  • Sirve dist/ como archivos estáticos
  • Redirige todas las rutas no-API a dist/index.html (enrutamiento SPA)
  • Gestiona /api/* para la orquestación de R2
ℹ️
La app es un servidor Node.js tradicional — despliega en cualquier VPS, contenedor o PaaS que soporte Node.js (Railway, Render, Fly.io, AWS EC2). No es compatible con entornos serverless sin adaptación.

Mejoras propuestas

UX / Producto

  • Cola de múltiples archivos (carga por lotes)
  • Restricciones de tipo de archivo (prop accept en DropZone)
  • Vista previa de miniaturas para imágenes y vídeos
  • Historial de subidas (localStorage)
  • Enlaces cortos compartibles

Fiabilidad

  • Reanudar cargas interrumpidas (persistir uploadId + key en sessionStorage)
  • Tamaño de fragmento y concurrencia configurables
  • Validación de tamaño de archivo en servidor
  • Webhook al completar la carga

Seguridad y operaciones

  • Autenticación (magic link, OAuth)
  • Rate limiting en endpoints de la API
  • URLs de descarga firmadas con expiración (en lugar de bucket público)
  • Reglas de ciclo de vida de objetos (auto-eliminar tras N días)
  • Integración de análisis de virus

Experiencia de desarrollador

  • Tests unitarios para useUpload, uploader.ts, format.ts
  • Tests E2E (Playwright / Cypress)
  • Configuración con Docker Compose
  • Pipeline CI/CD
  • CHUNK_SIZE y MAX_CONCURRENT configurables vía variables de entorno
